Detailed Description

The main interface for Audio Processors.

The IAudioProcessor interface is expected to be implemented by the audio decoding plugin class. On Android this should be made accessible to the app through the CreateAudioProcessor C function;

IAudioProcessor* CreateAudioProcessor(void);

On iOS, the calling app will need to create an IAudioProcessor-based class using new, and pass it down through the ISDecoder. The IAudioProcessor interface is declared in the IAudioProcessor.h C++ header file and inherits from the INamedObject interface.

virtual const IProcessorEventParams* intrasonics::IAudioProcessor::PrepareData ( const NTPInfo *  ntpInfo)
pure virtual

Gives the Audio Processor a chance to PrepareData.

Indicates to the Audio Processor that the AudioDecoder is expecting it to prepare any data it has collected for uploading to the server. Once the Audio Processor has prepared the data it should either report back the prepared data synchronously in the return value or report back through the IAudioProcessorListener::onProcessorEvent() method to indicate what kind of data is ready for upload.

Parameters
ntpInfoupdated NTP info acquired just before the call.
Returns
the event params describing the result of preparing data. If the implementer chooses to prepare the data asynchronously, this can be nullptr
Note
As AudioProcessors are expected to have different requirements for uploading data or returning events, this method is only called for audio processors which need to prepare data on a schedule. If an AudioProcessor requires a different schedule from this or passes data of a different nature, the AudioProcessor may choose to ignore this call and can call the IAudioProcessorListener::onProcessorEvent method whenever it chooses. In the standard SDK ntpInfo will always be null. In the standard SDK PrepareData will not be called and Audio Processors should implement event-based data communications.

Process a buffer of samples.

Process a buffer of samples. The samples are recorded in 16 bit PCM mono, at the rate supplied by the call to Initialise method. If a plugin requires samples in a different format it will need to convert them. If a different sample rate is required the plugin can implement the ISampleRateAdvisor sample rate to allow the SDK to query the sample rate that the plugin requires. The SDK will then attempt to provide audio buffers at that sample rate.

Parameters
samplesa buffer of samples to process
timeStampa monotonic timeStamp; see chrono::steady_clock::time_point for more details
numSamplesthe number of samples in the buffer
Note
The contents of the samples buffer are guaranteed only until the next time ProcessBuffer is called. Triple buffering is used to ensure that small delays in processing do not affect the overall running of the app. However, on average the AudioProcessor should ensure that it's processing of each buffer is completed before the next buffer arrives.

Starts the audio processor.

Start the processing of samples. After this call, the AudioProcessor will be sent sample buffers. This method gives the AudioProcessor a chance to initialise any structures it needs to start processing buffers.

virtual void intrasonics::IAudioProcessor::Stop ( )
pure virtual

Stop the audio processor.

Stop the processing of samples. No further samples will be sent to the AudioProcessor until Start is called. This method gives the AudioProcessor a chance to finish up whatever it has been doing if it needs to.

Note
Other calls to the IAudioProcessor can still occur after Stop is called. Only ProcessBuffer is guaranteed not to be called.
